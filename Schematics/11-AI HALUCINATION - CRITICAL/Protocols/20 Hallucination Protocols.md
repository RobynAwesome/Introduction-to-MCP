# 20 Hallucination Protocols

1. log the exact false claim before anything else
2. log the exact truth separately
3. identify the primary evidence source
4. state whether the claim came from memory, inference, haste, or fabrication
5. classify severity before justification
6. classify domain before excuse: software, hardware, process, hierarchy, memory, research, or role-mapping
7. classify fix ownership before proposing a solution
8. distinguish hallucination from omission, misunderstanding, and speculation
9. state whether code, docs, runtime, hierarchy, or planning was affected
10. record recurrence risk as `high`, `medium`, or `low`
11. forbid vague defensive wording
12. forbid claiming certainty without evidence
13. record whether the incident was preventable
14. record what check should have happened before the false claim
15. add the incident to the database master
16. update taxonomy if the type is new or refined
17. update the solutions library if the fix pattern is new or improved
18. extract the training lesson for Orch
19. if the incident affected team direction, add a lead-governance note too
20. once Master flags the hallucination, logging it here is mandatory
